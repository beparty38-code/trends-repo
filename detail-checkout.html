먼저, GitHub Actions란?

GitHub에 내장된 자동화 도구입니다. "코드를 push하면 자동으로 테스트 돌려줘", "PR 올리면 빌드 확인해줘" 같은 작업을 자동화합니다.

비유하면: GitHub Actions = "GitHub 안에 사는 로봇 비서"
당신이 규칙(워크플로우)을 정해주면, 이벤트가 발생할 때마다 자동으로 일을 처리합니다.

핵심 용어 5가지

용어뜻비유
Workflow자동화 규칙 파일 (.yml)레시피 (요리 순서)
Event워크플로우를 작동시키는 트리거"주문이 들어오면" (push, PR 등)
Job하나의 러너에서 실행되는 작업 묶음요리사 한 명의 업무
StepJob 안의 개별 명령요리의 각 단계
Runner워크플로우가 실행되는 가상 서버주방 (깨끗한 빈 공간)

워크플로우 파일 해부

모든 워크플로우는 .github/workflows/ 폴더에 YAML 파일로 작성합니다:

# .github/workflows/ci.yml

name: CI                        # 워크플로우 이름

on: [push, pull_request]        # 언제 실행? push 또는 PR 때

jobs:
  test:                          # Job 이름
    runs-on: ubuntu-latest       # 어떤 러너? (Ubuntu VM)

    steps:                       # 실행할 단계들
      - uses: actions/checkout@v7   # Step 1: 코드 가져오기
      - run: npm install            # Step 2: 의존성 설치
      - run: npm test               # Step 3: 테스트 실행
Tip: uses:는 남이 만든 액션을 쓰겠다는 뜻이고, run:은 직접 셸 명령을 실행하겠다는 뜻입니다.

왜 checkout이 필요한가?

GitHub Actions의 러너(Runner)는 완전히 빈 가상머신입니다. 여러분의 코드가 없습니다!

GitHub 서버 Runner (빈 VM) ┌─────────────┐ ┌─────────────────┐ │ 내 레포 │ │ │ │ ├── src/ │ │ (텅 비어있음) │ │ ├── test/ │ ───? │ │ │ └── ... │ │ 코드가 없으니 │ │ │ │ 아무것도 못 함 │ └─────────────┘ └─────────────────┘ actions/checkout 이 하는 일: ┌─────────────┐ ┌─────────────────┐ │ 내 레포 │ git │ $GITHUB_ │ │ ├── src/ │ ─fetch→│ WORKSPACE/ │ │ ├── test/ │ clone │ ├── src/ │ │ └── ... │ │ ├── test/ │ │ │ │ └── ... │ └─────────────┘ └─────────────────┘
checkout 없이 실행하면?
npm install → "package.json이 없습니다" 에러
python test.py → "파일을 찾을 수 없습니다" 에러
코드 자체가 VM에 없으니까요.

checkout이 실제로 하는 일 (내부 동작)

  1. 인증 설정 — GITHUB_TOKEN으로 레포에 접근 권한 확보
  2. git init + fetch — 러너에 빈 git 레포 만들고, GitHub에서 코드 가져오기
  3. ref 체크아웃 — 트리거된 브랜치/태그/커밋으로 이동 (기본: 이벤트의 SHA)
  4. 인증 토큰 저장 — 이후 step에서 git push 등을 할 수 있도록 설정
  5. 서브모듈 처리 — 옵션에 따라 서브모듈도 함께 가져오기
핵심: 기본적으로 fetch-depth: 1로 최신 커밋 1개만 가져옵니다 (shallow clone). 빠르지만, git log로 히스토리를 보려면 fetch-depth: 0이 필요합니다.

사용 패턴 모음

1. 기본 — 가장 많이 쓰는 형태

steps:
  - uses: actions/checkout@v7
  - run: npm test

2. 전체 히스토리가 필요할 때

버전 태깅, changelog 생성, git log 분석 등

- uses: actions/checkout@v7
  with:
    fetch-depth: 0    # 전체 커밋 히스토리

3. 특정 폴더만 가져올 때 (Sparse Checkout)

거대 모노레포에서 필요한 부분만 빠르게

- uses: actions/checkout@v7
  with:
    sparse-checkout: |
      .github
      packages/my-app

4. 다른 레포의 코드도 필요할 때

# 내 레포
- uses: actions/checkout@v7
  with:
    path: main

# 공용 도구 레포
- uses: actions/checkout@v7
  with:
    repository: my-org/shared-tools
    path: tools

5. PR에서 자동 커밋 push

린트 자동 수정, 포맷팅 등을 자동 커밋

- uses: actions/checkout@v7
  with:
    ref: ${{ github.head_ref }}    # PR의 브랜치
- run: npx prettier --write .
- run: |
    git config user.name "github-actions[bot]"
    git config user.email "github-actions[bot]@users.noreply.github.com"
    git commit -am "style: auto-format" || true
    git push

6. SSH 키로 Private 레포 접근

- uses: actions/checkout@v7
  with:
    repository: my-org/private-repo
    ssh-key: ${{ secrets.DEPLOY_KEY }}

주요 설정 옵션

Input기본값설명
repository현재 레포체크아웃할 레포 (owner/repo)
ref이벤트 ref브랜치/태그/SHA 지정
tokengithub.token인증 토큰
ssh-key-SSH 키 인증 (private 레포)
fetch-depth1커밋 깊이 (0 = 전체)
fetch-tagsfalse태그 포함 여부
sparse-checkout-특정 경로만 가져오기
submodulesfalse서브모듈 처리 (true/recursive)
lfsfalseGit LFS 파일 다운로드
cleantrue기존 파일 정리 (git clean)
persist-credentialstrue이후 step에서 git push 가능
path.체크아웃 위치 (멀티 레포 시 사용)

보안 설계

pwn request 방지

악의적인 Fork PR이 워크플로우를 변조해 시크릿을 탈취하는 공격을 방지합니다.

기본 동작: Fork PR의 코드 체크아웃은 거부됩니다.
allow-unsafe-pr-checkout: true를 명시적으로 설정해야만 허용됩니다.

자격증명 보안

v6부터 토큰은 $RUNNER_TEMP 아래 별도 파일에 격리 저장됩니다. 워크플로우가 끝나면 자동 삭제.

권장 권한 설정

permissions:
  contents: read    # 최소 권한 원칙

GitHub Actions 실전 예제 모음

Node.js 앱 CI

name: Node CI
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20, 22]    # 여러 Node 버전에서 테스트
    steps:
      - uses: actions/checkout@v7
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm ci
      - run: npm test

Python 린트 + 테스트

name: Python CI
on: push

jobs:
  lint-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v7
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: pip install -r requirements.txt
      - run: ruff check .
      - run: pytest

자동 배포 (GitHub Pages)

name: Deploy
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      pages: write
      id-token: write
    steps:
      - uses: actions/checkout@v7
      - run: npm ci && npm run build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: dist/
      - uses: actions/deploy-pages@v4

Project Structure

actions/checkout/
├── src/              # TypeScript 소스 코드 (핵심 로직)
├── __test__/         # Jest 테스트
├── adrs/             # Architecture Decision Records
├── dist/             # 빌드된 배포 파일 (git에 포함!)
├── action.yml        # 액션 메타데이터 (inputs/outputs 정의)
└── package.json
특이점: dist/ 폴더가 git에 포함됩니다. GitHub Actions 런타임이 node_modules를 설치하지 않고 바로 실행하기 때문에, 번들된 결과물을 레포에 커밋해야 합니다.

Learn Points

  • GitHub Actions 액션 개발 — action.yml로 inputs/outputs를 선언하고, TypeScript로 로직 구현
  • Git 내부 동작 — fetch, shallow clone, sparse checkout이 어떻게 동작하는지
  • 보안 설계 — pwn request 방지, 자격증명 관리, 최소 권한 원칙
  • ADR 패턴 — 아키텍처 결정을 문서로 남기는 방법
  • @actions/* 패키지 — core, exec, github 등 Actions SDK 활용
